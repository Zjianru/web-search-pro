# Web Search Pro 2.1.0 Release Pack

This file contains release materials for both GitHub and ClawHub.

## Release Summary

`Web Search Pro 2.1.0` is the first feature release after the `2.0` retrieval-stack reset.

The core direction does not change: this is still a retrieval system for agents and upstream
models, not just a single search script. What changes in `2.1.0` is the strength of the search
substrate itself:

- a richer search router with structured signals and compact route transparency
- a broader provider layer with `Brave`, `Querit`, `You.com`, `SearXNG`, and Perplexity / Sonar
- an explicit `bootstrap` contract for agents
- head-to-head benchmark coverage against `web-search-plus`
- synchronized ClawHub package copy and compliance posture

Product version is now `2.1.0`.  
Stable JSON schema remains `1.0`.

## GitHub Release Notes

### Title

`v2.1.0 - Upgrade the search and routing substrate`

### Body

`Web Search Pro 2.1.0` sharpens the search and routing layer on top of the existing `2.0`
retrieval stack.

This release is about making the search substrate more competitive and more honest for agent use:
better provider coverage, explainable route selection, cache visibility, agent bootstrap, and
benchmarkable claims.

### Highlights

- Added `Brave`, `Querit`, `You.com`, `SearXNG`, and `Perplexity / Sonar` support
- Added gateway-backed Perplexity transport paths for:
  - native Perplexity
  - OpenRouter
  - Kilo
  - custom OpenAI-compatible gateways
- Upgraded the planner into `search-router v2` with:
  - structured query signals
  - `selectionMode`
  - `confidence`
  - `topSignals`
  - richer `routingSummary` and routing diagnostics
- Added `bootstrap.mjs` as an agent-readable runtime contract
- Exposed cache hit and TTL telemetry directly in retrieval outputs
- Added `head-to-head` and `head-to-head-live` benchmark suites against `web-search-plus`

### Why This Release Matters

Before `2.1.0`, `web-search-pro` was already broader than search-router-first competitors because
it had `extract`, `crawl`, `map`, `research`, diagnostics, and review surfaces.

The main gap was the search substrate itself:

- provider coverage
- search intent routing
- route transparency
- comparable benchmark evidence

`2.1.0` closes much of that gap without abandoning the wider retrieval-stack architecture.

### Compatibility

- Product version is now `2.1.0`
- Stable machine-readable JSON schema remains `1.0`
- Existing `search`, `extract`, `crawl`, `map`, and `research` consumers remain valid
- New fields such as `routingSummary.selectionMode`, `routingSummary.confidence`,
  `routingSummary.topSignals`, `cached`, and `cache` remain additive

### ClawHub And Compliance

- ClawHub publishing still uses the generated core profile, not the repository root
- Generated package metadata and docs now match the broader provider set more closely
- `node` remains the only hard runtime requirement
- Optional provider envs remain optional and explicitly disclosed
- Review-visible runtime surfaces now include `bootstrap.mjs` alongside `capabilities.mjs`,
  `doctor.mjs`, and `review.mjs`

## ClawHub Release Description

`Web Search Pro 2.1.0` upgrades the core retrieval stack with a stronger search substrate for
agents.

This release adds:

- broader search-provider coverage
- explainable route selection with confidence and top signals
- an agent-readable bootstrap contract
- cache telemetry for agents
- benchmark coverage against `web-search-plus`

The generated ClawHub package still keeps the registry-facing install profile narrow:

- search
- extract
- crawl
- map
- research
- bootstrap / doctor / capabilities / review
- cache / health

through Tavily, Exa, Querit, Serper, Brave, SerpAPI, You.com, SearXNG, and Perplexity / Sonar.

## ClawHub Maintainer Compliance Note

Maintainer note: `web-search-pro 2.1.0` keeps the current two-surface distribution strategy on
purpose.

Why:

- the full repository contains the complete local source tree, including render, eval, and tests
- the ClawHub registry should review a narrower installed artifact
- optional provider credentials still do not map cleanly to a single hard metadata gate

For that reason:

- GitHub remains the canonical full source tree
- ClawHub continues to publish the generated core profile
- metadata still declares only the real hard runtime requirement: `node`
- optional env disclosure is present, but remains optional
- runtime truth remains visible through `capabilities.mjs`, `doctor.mjs`, `bootstrap.mjs`, and
  `review.mjs`

The generated package now reflects the current provider and routing posture more accurately,
including:

- `Querit` and `Brave`
- gateway-backed Perplexity / Sonar access
- the agent bootstrap surface
- route transparency and cache visibility in the installed runtime
